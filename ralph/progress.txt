## Codebase Patterns
- vitest.config.ts must NOT import vite.config.ts directly — vue-devtools plugin uses localStorage which breaks in Node. Define plugins/aliases separately.
- `bun run test:app` runs on host, not Docker
- tsconfig.app.json already excludes `src/**/__tests__/*`, tsconfig.node.json already includes `vitest.config.*`
- Pre-existing typecheck failures exist (google namespace, API module resolution) — not caused by test infra
- Test helper `mountWithPlugins` creates fresh i18n + Pinia per test — import from `@/__tests__/helpers/setup`
- Module-level shared state (like `alerts` ref in useAlert) must be cleaned in beforeEach — use while loop, not forEach, to avoid mutate-during-iterate
- Importing pinia/auth store triggers vue-devtools `localStorage.getItem` at import time — mock `@/stores/auth` with `vi.mock` instead of using real pinia for composable tests that depend on auth store
- Composables using Vue lifecycle hooks (onMounted/onUnmounted) need a component wrapper — use `mount(defineComponent({ setup() { ... } }))` from @vue/test-utils
- Don't use `mountWithPlugins` when pinia is not needed — it triggers the vue-devtools localStorage crash
- Mock `vue-router` with `vi.mock` to capture `onBeforeRouteLeave` callback for testing route guards
- `window.confirm` doesn't exist in happy-dom — assign `window.confirm = vi.fn()` in beforeEach
- Unmount wrappers in afterEach to prevent beforeunload listener leaks across tests
- Mock `hono/client` with nested Proxy to intercept `client[endpoint][$method]()` calls — `hc` returns Proxy→Proxy→mockFn
- Store tests need `vi.stubGlobal("localStorage", ...)` with a Map-backed mock — happy-dom/Node's built-in localStorage lacks proper Storage API methods
- Use `vi.resetModules()` in beforeEach + dynamic `import("pinia")` and `import("../store")` to get fresh store instances with fresh localStorage state
- Pinia store tests: set localStorage BEFORE `getStore()` to test hydration; the store factory reads localStorage at creation time
- Use `vi.hoisted()` to declare mock fns referenced inside `vi.mock` factories — avoids "cannot access before initialization" error
- For API composable tests: mock `hono/client`, `@/stores/auth`, `vue-i18n`, `@/composables/utils/useAlert`, and partially mock `@/utils/api-error` (keep real ApiError, mock parseApiError)
- Mock `@/composables/useTauri` with mutable `mockIsTauri = { value: true }` for stores that depend on Tauri — controls localStorage gating per test
- Atom components don't use i18n/pinia — use plain `mount` from @vue/test-utils, not `mountWithPlugins`
- Molecule components using `@vueuse/core` (e.g. useMediaQuery) — mock the module with `vi.mock("@vueuse/core", ...)`
- DataTable has no built-in loading prop — "loading state" is managed externally by the parent component
- Stub child components (SortIcon, TruncateWithTooltip) when testing molecule components — use functional stubs with `<slot />` to preserve content rendering
- Dialog uses Teleport — stub with `global: { stubs: { Teleport: true } }` for testing
- Dialog watch on modelValue is NOT immediate — mount with `false` then `setProps({ modelValue: true })` to trigger keydown listener registration
- Components using `$t()` but not pinia: create `createI18n()` manually and pass as plugin — avoids pinia/vue-devtools crash while providing i18n
- Use `button[title="..."]` with i18n strings to find specific buttons — more reliable than class-based group/index selection
